one macOS fileless malware analyze

发表于 2020-01-30

背景

fileless，无文件攻击，在windows上经常出现的一种攻击方式。意思是在一个恶意攻击过程中，执行攻击的载荷不会以文件的形式出现在计算机硬盘上，而是直接在内存进行加载执行。大部分都是以脚本命令的形式在内存里直接解释执行，也有的是恶意样本自己执行内存映射加载一段在内存里的文件数据，进而执行进行恶意操作。前一段时间有注意看到相关的信息，最近比较闲，所以分析记录一下这个macOS上相关的恶意攻击。

样本

UnionCryptoTrader.dmg

UnionCryptoTrader.dmg

sha1:06d9f835efd1c05323f6a3abdf66e6be334e47c4

样本来源

VT检测结果


看上去目前大部分厂商都可以检测到了，但是闲着无聊还是分析一下样本的主要行为。

分析

样本为一个dmg文件，加载以后获得一个pkg包，对pkg文件解压

可以发现pkg是用来安装UnionCryptoTrader.app ，文件没有厂商信息，在安装结束以后会执行postinstall脚本。查看一下postinstall脚本内容

脚本的功能就是拷贝一个plist文件和一个可执行文件到对应目录，设置自启动unioncryptoupdater

可以想到主要的行为都在这个自启动的文件中进行。
unioncryptoupdater
程序开始时，先获取本地系统deviceSerial 和OSVersion，然后将这部分信息作为请求的一部分发送给服务器。



发送数据请求：

将这些数据发送给server之后，接受返回的结果到。另外可以看到此次发送请求中act 是check，检查sever能否连接到以及要进行的操作，如果返回错误码不是200或者返回结果为字符串“0”，会经过一段休眠之后会再次尝试连接。

如果返回结果为要处理的数据，则会使用base64解码，然后函数processUpdate执行数据的操作。

从processUpdate的实现，可以看出解码后的数据其实是一个可执行文件，首先会尝试在内存中加载执行，如果加载失败则会使用写入文件然后执行文件的常规方法。

load_from_memory就是实现fileless的函数，首先将数据以可读可写可执行权限匿名映射到共享内存，然后调用memory_exec2 执行。

由于内存中过程映像的布局与磁盘上映像的布局不同，因此不能简单地将文件复制到内存中并直接执行它。相反，必须调用诸如NSCreateObjectFileImageFromMemory和NSLinkModule之类的API（它们负责准备内存中的映射和链接）。所以memory_exec2便是用了这些API，然后查找macho文件的magic number：0xfeedfacf （MH_MAGIC_64）。找到macho文件之后再去遍历找到可执行文件结构中的LC_MAIN，最终获取入口地址，进行调用，获取返回值。

总结

这个算是一个比较简单的样本，具有在内存中下载和执行macho文件的功能，之前是没有见到的。虽然由于样本应用程序没有有效签名，在执行的时候系统会进行反馈请求授权，但是用户可能会因为需要使用应用程序而同意授权。

mach-O file format

发表于 2019-07-07 更新于 2020-01-30

mach-O 文件是macOS上的二进制文件格式，相当于windows上的PE格式 跟 linux上的elf文件格式，macOS上的主要的可执行文件都是这种格式，今天就分析下mach-O的文件格式

Format

mach-O文件主要分为下面三部分:

  • mach header
  • load commands
  • segments(Data)

如上图所示。同时在xnu中也有对于这种格式的一些结构定义，下面从结构上分析一下。

macho header

macho header中保存了文件的magic、cputype 、文件大小等一些基本信息。对于更细节的信息，可以参考xnu 源代码中，对于macho header的定义。

主要参数:

magic:可以文件的标志，可以识别出文件是32位还是64位 同时也可以区分文件时大端还是小端\
cputype:标识cpu的架构，比如i386、x86_64、arm等\
filetype:下面列出一些主要的filetype

FILETYPE a
MH_OBJECT 编译过程产生的目标文件 .o
MH_EXECUTE 可执行文件
MH_CORE core dump
MH_DYLIB 动态库
MH_KEXT_BUNDLE 内核扩展文件
MH_DYLINKER 调试符号文件

Load commands

紧接着header 后面就是load commands的部分，由n个load_command的结构组成，数量跟大小在header里面可以获取到。

对于不同类型的load_command 可以用不同的结构体来读取后面的数据，主要的一些类型:
LC_SEGMENT
LC_UNIXTHREAD unix 线程
LC_MAIN 主线程
LC_UUID 唯一ID
LC_LOAD_DYLINKER 连接器
LC_LOAD_DYLIB 依赖的动态库

ps:从Mountain Lion开始，一条新的加载命令LC_MAIN替代了LC_UNIXTHREAD命令。这条命令的作用是设置程序主线程的入口点地址和大小。这条命令比LC_UNIXTHREAD命令更实用一个些，所以后来的macho 不一定兼容之前的系统。

最多的结构是LC_SEGMENT

segment_command 结构体中保存着结构体的基本信息，包括在文件中偏移地址,占用文件大小，虚拟地址，虚拟地址大小，用于解析macho文件，像一个索引一样，真正的指向的数据在下一部分的data部分。

Data & Segment

data 部分分为不同的segment 其中segment 可能包含多个section，是存放主要数据的地方，前面segment的数据都指向这里。section的结构会帮助内核将数据映射到虚拟内存。
__text 代码
__data the real initialized data section
__cstring 硬编码的字符串
- -

小结

理解macho的结构，便于后面理解macho文件在系统的加载过程，同时对于一些样本分析 以及相对底层的安全开发也有帮助。
